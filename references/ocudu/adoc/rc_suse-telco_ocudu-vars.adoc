// TECHNICAL COMPONENTS
//   Identify the technical components featured in the guide.

//
//   Variables are especially useful for referring to various
//   aspects of the solution components, such as properly written names,
//   product/project versions, website URLs, and so on.
//   Keep in mind the following best practices for variable names:
//     1. Be consistent.
//     2. Prefer recognizable variable names over generic ones.
//        E.g.: 'sles' and 'srancher' are better variable
//        names than 'comp1' and 'component2'.
//     3. Prefer short variable names over long ones.
//        E.g.: 'sles' is better than 'suselinux', and
//        'svirt' is better than 'susevirtualization'.
//     4. Consider following a simple pattern to associate related variables,
//        such as `base[-modifier]'.
//        E.g.: 'svirt-brand', 'srancher-docs', 'sles-version'.
//        Some standard '-modifiers' could include:
//         -provider: Name of company or project providing the component
//            E.g.: 'SUSE', 'HPE', 'Microsoft', 'Veeam'
//         -long: Full name of the product without trademark symbols
//            E.g.: 'SUSE Rancher Prime', 'Portworx Enterprise by Everpure'
//         -brand: Branded, full name of the component
//            E.g.: 'SUSE® Virtualization', 'NetApp® Trident™'
//         -version: The relevant product version referenced in
//            the document
//            E.g.: '15 SP7', '3.2.4'
//            If additional product versions are called out in the
//            document, you could store them in separate variables.
//            E.g.: 'k3s-version1', 'k3s-version2')
//         -website: Product website
//            E.g.: 'https://www.suse.com/products/server/'
//         -docs: Product documentation
//            E.g.: 'https://documentation.suse.com/sles/'
//
// IMPORTANT: The templates leverage generic variable names, such as
//            'comp1' and 'comp2-version1'.
//            Be sure to replace these generic variable names with your
//            own variables wherever they occur, including:
//            in the Vars file, the Docinfo file, and your Main content file.
