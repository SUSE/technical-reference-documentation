== Introduction

// = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = =
// Introduction
// = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = =

In this guide, you learn how to leverage {ontap-brand} and the {trident-brand} to provide to deliver high-performance {ontap-provider} storage for virtual machines (VMs) hosted in your {svirt-brand} environment.

////
This guide focuses on the solution comprising SUSE Virtualization and NetApp ONTAP Storage, leveraging Trident Container Storage Interface (CSI).
The motivation and purpose of this solution center around enabling SUSE Virtualization to use external, high-performance NetApp storage for persistent volumes, specifically allowing for the creation of VMs with disks assigned using a NetApp-backed StorageClass.
////


=== Scope

This guide covers the detailed steps required for setting up a three-node {svirt-long} cluster to consume {ontap-provider} storage with the {trident-long}.
It includes steps for LUN provisioning, multipath configuration, {trident} installation, StorageClass definition, and basic validation.


=== Audience

This guide is intended for IT professionals responsible for virtualization and infrastructure management, such as infrastructure architects, systems administrators, or platform engineers.

To successfully follow this guide, you should have:

* Familiarity with {svirt-long} deployment and configuration
* Experience with {ontap-provider} storage provisioning and management
* Understanding of iSCSI concepts and configuration
* Working knowledge of Kubernetes

=== Prerequisites

The following resources and configurations are required to support the solution:

* **{svirt-long} Cluster:** A functional {svirt-docs}[{svirt-long}] cluster must be installed and operational.
* **{ontap-long}:** A {ontap-docs}[NetApp ONTAP] system must be available that meets the Trident environment prerequisites and is accessible from the {svirt-long} cluster nodes.
* **Network Connectivity:** Robust network connectivity between all {svirt-long} nodes and the NetApp ONTAP system is required to support storage operations.
* **Sufficient Resources:** {svirt-long} nodes must have adequate  {svirt-docs-reqs}[CPU, memory, and local disk space] to support the Trident deployment and the expected workloads. Ensure the environment meets the minimum requirements for a stable and supported deployment. A Trident-managed Kubernetes storage setup must include at least three nodes, each meeting specific hardware, software, and network requirements.
* **Administrative Access:** You need administrative access to the {svirt} UI and the underlying Kubernetes cluster to apply configurations and manage resources.
See {svirt-docs-faq}[{svirt}: Troubleshooting FAQ].
* **Trident and CSI Driver Versions:** Use {trident-docs}[NetApp Trident] 23.x or newer and a compatible CSI driver version.

== Architectural overview

// = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = =
// Architecture
// = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = = =

The diagram below illustrates the basic architecture for integrating NetApp storage with {svirt-long}.

image::architecture_diagram.png[title="Architecture: SUSE Virtualization with NetApp ONTAP and Trident", scaledwidth="90%", align="center"]

The solution consists of the components identified below:

* **{svirt-long} (3 nodes):** Provides compute and orchestration for VMs and containers.
* **{ontap-long}:** External storage backend providing iSCSI LUNs.
* **{trident-long}:** Kubernetes CSI driver enabling dynamic provisioning of persistent storage.

**Logical Flow:** {svirt-long} nodes connect to the NetApp ONTAP system over the iSCSI protocol. Multipath provides high availability and redundancy. Trident CSI manages storage lifecycle operations, dynamically provisioning volumes for virtual machines.

== Deployment overview

This section provides practical guidance for deploying a functional proof-of-concept implementation of the solution with additional considerations for scaling to production.

For deployment purposes, the solution can be illustrated as a layered stack. The deployment process then consists of installing and configuring each layer of this stack, typically from bottom (compute platform) to top (software applications). The following sections provide guidance for deploying each layer of the solution.

=== Compute platform deployment

The compute platform provides the foundational layer for the solution. The following steps detail how to prepare the {svirt-long} nodes for iSCSI integration.

*Preparation*

. Install a {svirt-long} high-availability cluster.
.. Ensure each node meets minimum hardware requirements and that hardware virtualization (VT-x/AMD-V) is enabled.
.. Deploy {svirt-long}. See the official documentation for detailed guidance.
.. After deployment, confirm you have access to the {svirt-long} cluster management UI and `kubeconfig`.

*Process*

. Configure iSCSI Initiator (Persistent Configuration via CloudInit YAML).
.. Create a CloudInit resource to auto-start each node. You must include the following configuration in `/oem`.
+
[source, yaml]
----
apiVersion: node.harvesterhci.io/v1beta1
kind: CloudInit
metadata:
  name: iscsi-ip-config
spec:
  matchSelector: {}
    kubernetes.io/hostname: harvester-node-1 # Replace with actual hostname
  filename: 01_iscsi_ip.yaml
  contents: |
    stages:
      initramfs:
        - name: "Configure iSCSI network interfaces"
          files:
            - path: /etc/NetworkManager/system-connections/iscsi-path1.nmconnection
              owner: 0
              group: 0
              permissions: 0600
              content: |
                [connection]
                id=iscsi-path1
                type=ethernet
                interface-name=eno3np0
                autoconnect=true
                [ipv4]
                method=manual
                addresses=192.168.100.87/24 # Replace the IP address
                never-default=true
            - path: /etc/NetworkManager/system-connections/iscsi-path2.nmconnection
              owner: 0
              group: 0
              permissions: 0600
              content: |
                [connection]
                id=iscsi-path2
                type=ethernet
                interface-name=eno4np1
                autoconnect=true
                [ipv4]
                method=manual
                addresses=192.168.200.87/24 # Replace the IP address
                never-default=true
----
+
.. Apply the configuration on each node:
+
[source, console]
----
kubectl apply -f 01_iscsi_ip.yaml
----
+
.. Verify the status of the CloudInit resource:
+
[source, console]
----
kubectl get cloudinit
----

. Enable and Persist iSCSI Service
+
Create a CloudInit resource to auto-start the iSCSI daemon, `iscsid`:
+
[source, yaml]
----
apiVersion: node.harvesterhci.io/v1beta1
kind: CloudInit
metadata:
  name: iscsid-start
spec:
  matchSelector: {}
  filename: 99_iscsid.yaml
  contents: |
    stages:
      network:
        - name: "Enable and start iscsid"
          systemctl:
            enable:
              - iscsid
            start:
              - iscsid
----
+
Apply on any node:
+
[source, console]
----
kubectl apply -f 99_iscsid.yaml
----
+
Verify status of `iscsid`:
+
[source, console]
----
systemctl status iscsid
----

. Provision a LUN from NetApp Storage and Map It to Cluster
.. Log in to the NetApp ONTAP System Manager.
.. Create a new LUN and assign it to an iSCSI initiator group containing the {svirt-long} nodes' IQNs.
.. Note the following information:
* LUN path
* Target IQNs
* iSCSI portal IPs

. Enable and Configure Multipath
.. Create a CloudInit resource to auto-start the multipath daemon (`multipathd`) with the loaded configuration:
+
[source, yaml]
----
apiVersion: node.harvesterhci.io/v1beta1
kind: CloudInit
metadata:
  name: multipathd-config
spec:
  matchSelector: {}
  filename: 99_multipathd.yaml
  contents: |
    stages:
      network:
        - name: "Configure multipath.conf"
          files:
            - path: /etc/multipath.conf
              permissions: 0644
              owner: 0
              group: 0
              content: |
                defaults {
                    find_multipaths no
                    user_friendly_names yes
                }
                blacklist {
                    device {
                        vendor ".*"
                        product ".*"
                    }
                }
                blacklist_exceptions {
                    device {
                        vendor "NETAPP"
                        product "LUN"
                    }
                }
        - name: "Enable and start multipathd"
          systemctl:
            enable:
              - multipathd
            start:
              - multipathd
----
+
.. Apply on any node:
+
[source, console]
----
kubectl apply -f 99_multipathd.yaml
----
+
.. Verify status of cloudinit and multipathd:
+
[source, console]
----
kubectl get cloudinit
systemctl status multipathd
----

=== Software

The following sections provide deployment guidance for the software layer, specifically the installation and configuration of {trident-long}.

==== Trident CSI Deployment

*Preparation*

. Prepare Worker Nodes
+
Before installing {trident-long}, ensure that each {svirt-long} node meets the prerequisites for Kubernetes storage integration. Follow the official NetApp documentation for {trident-docs-worker-prep}[Worker Node Preparation for Trident CSI].

. Configure NetApp Storage Controller Secrets
+
Create a Kubernetes secret for ONTAP credentials:
+
[source, console]
----
kubectl create secret generic ontap-secret -n trident \
  --from-literal=username=admin \
  --from-literal=password='YourPassword'
----

*Process*

. Install {trident-long}
+
It is recommended to use `tridentctl` for installation. You can find more details in the {trident-docs-install}[NetApp Trident Installation Guide].
+
[source, console, subs="attributes+"]
----
wget {trident-installer-url}
tar -xzf trident-installer-24.06.0.tar.gz
cd trident-installer
./tridentctl install -n trident
----
+
Verify that all Trident pods are in the `Running` state:
+
[source, console]
----
kubectl get pods -n trident
----

. Configure Trident Backend
+
Create a backend configuration file (e.g., `ontap-backend.json`):
+
[source, json]
----
{
  "version": 1,
  "storageDriverName": "ontap-san",
  "managementLIF": "10.10.10.50",
  "dataLIF": "192.168.100.10",
  "svm": "harvester_svm",
  "username": "admin",
  "password": "YourPassword",
  "igroupName": "harvester_igroup"
}
----
+
Apply the backend:
+
[source, console]
----
./tridentctl create backend -f ontap-backend.json -n trident
----
+
Validate the backend:
+
[source, console]
----
./tridentctl get backend -n trident
----

. Define StorageClass and SnapshotClass
+
Create a file named `netapp-storageclass.yaml`:
+
[source, yaml]
----
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: netapp-storageclass
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: csi.trident.netapp.io
parameters:
  fsType: ext4
  backendType: "ontap-san"
reclaimPolicy: Delete
volumeBindingMode: Immediate
allowVolumeExpansion: true
----
+
Apply it:
+
[source, console]
----
kubectl apply -f netapp-storageclass.yaml
----
+
Create a file named `netapp-snapshotclass.yaml`:
+
[source, yaml]
----
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: netapp-snapclass
driver: csi.trident.netapp.io
deletionPolicy: Delete
----
+
Apply it:
+
[source, console]
----
kubectl apply -f netapp-snapshotclass.yaml
----

. Register CSI Driver in {svirt-long} UI
.. Log in to the {svirt-long} UI.
.. Go to **Advanced > Settings > csi-driver-config**.
.. Click **Edit Setting**.
.. Set the provisioner to `csi.trident.netapp.io`.
.. Save changes.

. Set NetApp CSI Driver as Default StorageClass
+
Ensure the NetApp storage class is default and disable others if necessary:
+
[source, console]
----
kubectl patch storageclass harvester-longhorn \
  -p '{"metadata":{"annotations":{"storageclass.kubernetes.io/is-default-class":"false"}}}'

kubectl patch storageclass netapp-storageclass \
  -p '{"metadata":{"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
----

== Validation

Ensure that {trident-long} is correctly integrated with {svirt-long} by provisioning a virtual machine (VM) that uses a StorageClass backed by the NetApp ONTAP iSCSI backend.

. Create a VM
.. Log in to the {svirt-long} UI.
.. Navigate to **Virtual Machines** and click **Create**.
.. Select a namespace for your VM.
.. Provide a name for your VM.
.. On the **Basics** tab, configure CPU (e.g., 2) and Memory (e.g., 4 GB).
.. On the **Volumes** tab, select or add disks:
* For the root disk, select the StorageClass `netapp-storageclass` and specify the desired size.
* (Optional) Click **Add Disk** to add an ISO image.
.. Click **Create** to deploy the VM.

. Verify with the {svirt-long} UI
.. Navigate to **Virtual Machines > [Your VM] > Volumes**.
.. Confirm that the attached disk is using the `netapp-storageclass` StorageClass.

. Verify with the CLI
.. Log in to the {svirt-long} management node CLI as root.
.. List the PersistentVolumeClaims (PVCs) in the VM’s namespace:
+
[source, console]
----
kubectl get pvc -n <namespace>
----
+
.. Confirm that the **STORAGECLASS** column shows `netapp-storageclass`.

== Summary

Organizations are accelerating their digital transformation by modernizing applications and infrastructure — transitioning from traditional, virtual machine–centric environments to modern, cloud-native platforms powered by Kubernetes. {svirt-long} delivers a unified and flexible platform that allows organizations to seamlessly run both virtual machine (VM) and container workloads side-by-side on the same infrastructure.

{trident-long} extends this capability by providing enterprise-grade, dynamic storage provisioning for Kubernetes and {svirt-long} clusters. Through its integration with {ontap-long}, Trident enables resilient, scalable, and efficient iSCSI-based storage for mission-critical workloads, ensuring consistent performance, automation, and data protection across hybrid environments.

This guide detailed the steps to configure and validate the integration of {svirt-long} with {trident-long} and {ontap-long}, showcasing how enterprises can build a consistent, automated, and production-ready virtualization environment.
